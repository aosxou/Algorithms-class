## ⏱ 시간복잡도와 알고리즘 설계

> 🚨 **출제 포인트:** 빅오는 **최악의 경우(Worst Case)** 기준. 차수가 큰 항만 남기기. for문 개수로 O(n) 계산. 프로그램 예제 시간복잡도 직접 계산.

### 빅오(Big-O) 표기법 핵심 규칙

- **최악의 경우(Worst Case)** 기준으로 표기
- 상수 계수 및 **낮은 차수 항은 전부 제거** → 차수가 가장 큰 항만 남긴다
  - 예) `3n³ + 2n² + 5` → **O(n³)**
- **for문 1개** → `O(n)` / **for문 중첩 2개** → `O(n²)` / **중첩 3개** → `O(n³)`
- 입력 계수가 **지수로 올라가면 급격히 증가**함을 이해

### 빅오 복잡도 순위 (빠른 순 → 느린 순)

| 순위 | 표기법 | 명칭 | 비고 |
|:---:|:---:|:---:|:---:|
| 1 | O(1) | 상수 | ✅ 가장 빠름 |
| 2 | O(log n) | 로그 | |
| 3 | O(n) | 선형 | |
| 4 | O(n log n) | 선형로그 | |
| 5 | O(n²) | 이차 | |
| 6 | O(n³) | 삼차 | |
| 7 | O(2ⁿ) | 지수 | 🚨 가장 느림 |

> 아래로 갈수록 그래프가 **가파르게 올라감** → 알고리즘 성능 분석 실행 빈도수 표 참고

### 실행 빈도수 계산 예시

```c
// for문 1개 → O(n)
for(int i=0; i<n; i++) { ... }

// for문 중첩 2개 → O(n²)
for(int i=0; i<n; i++)
  for(int j=0; j<n; j++) { ... }

// for문 중첩 3개 → O(n³)
for(int i=0; i<n; i++)
  for(int j=0; j<n; j++)
    for(int k=0; k<n; k++) { ... }
```

## 📗 CH01 · 자료구조 소개

### 자료구조 형태별 분류 표 (괄호/빈칸 출제)

| 분류 | 종류 | 장점 | 단점 |
|:---:|:---|:---|:---|
| **단순 구조** | 정수, 실수, 문자, 문자열 | 기본 내장, 빠름 | 복합 데이터 표현 불가 |
| **선형 구조** (순차) | 배열, 스택, 큐, 덱 | 인덱스로 O(1) 빠른 접근 | 크기 고정, 삽입/삭제 느림 |
| **선형 구조** (연결) | 연결 리스트 | 동적 크기, 삽입/삭제 빠름 | 임의 접근 느림, 포인터 공간 낭비 |
| **비선형 구조** | 트리, 그래프 | 계층/관계 표현 가능 | 구현 복잡 |
| **파일 구조** | 순차, 색인, 직접 파일 | 대용량 데이터 처리 | I/O 속도 한계 |

> 💡 **순차 vs 연결 장단점**은 반드시 암기 (빈칸 채우기 출제)

---

### 자료의 표현 — 존(Zone) 형식 vs 팩(Pack) 형식

| 구분 | 존(Zone) 형식 | 팩(Pack) 형식 |
|:---:|:---|:---|
| **개념** | 사람이 읽기 위한 표현 (BCD 코드 활용) | 컴퓨터 연산을 위해 압축한 표현 |
| **예시** | `237` → `0010 0011 0111` (직접 연산 불가) | 존 형식을 압축하여 저장 (연산 가능) |
| **특징** | ❌ 메모리 낭비 심함 | ✅ 낭비 줄임 (pack = 압축) |

> 💡 **비유:** "백원"을 "back won"으로 쓰면 미국인은 못 읽음  
> → 사람을 위한 표현(존)과 기계를 위한 표현(팩)은 다름  
> → `237` → `010 011 111` 처럼 숫자로 표현하면 덧셈/뺄셈이 안 됨

---

### 정수 표현 — 보수(Complement) 방식

- **보수를 쓰는 이유:** 음수를 양수처럼 **덧셈 회로만으로 뺄셈 처리**
  - `A + (-B)의 보수 = A - B`
- **1의 보수:** 모든 비트 반전
- **2의 보수:** 1의 보수 + 1 (가장 많이 사용)

#### 🔑 2의 보수만 쓰는 이유 (반드시 암기)

```
  0 1 1 1 1 1 1 1  ← +127
  1 0 0 0 0 0 0 0  ← -128 (2의 보수에서만 가능)

  1의 보수: -127 ~ 0 ~ +127  (0이 두 개: +0, -0 존재 → 비효율)
  2의 보수: -128 ~ 0 ~ +127  (0이 하나, 표현 값 1개 더 많음 → 효율적!)
```

- **2의 보수**: 0의 표현이 유일하고, 표현 가능한 값의 범위가 1개 더 많음
- **표현 범위:** `-2^(n-1)` ~ `0` ~ `2^(n-1) - 1`

---

### 실수 — 부동 소수점(Floating Point)

- **사용 이유:** 미세하고 세밀한 값까지 저장하기 위해 사용
- 고정 소수점은 표현 범위가 좁음 → 부동 소수점으로 넓은 범위 + 높은 정밀도 표현
- **구성:** 부호 비트 + 지수부(Exponent) + 가수부(Mantissa)

---

### 문자 자료 — 각 코드별 특징

| 코드 | 특징 |
|:---:|:---|
| **BCD** | 6비트, 64가지 표현. 초기 컴퓨터 사용 |
| **ASCII** | 7비트(128가지). 최상위 비트 = **패리티(Parity) 비트** (오류 검출용). 미국 표준(American Standard Code for Information Interchange). 영어 중심 |
| **EBCDIC** | 8비트(256가지). IBM 메인프레임용 |
| **유니코드** | 16비트 이상. ASCII 한계 극복. **다국어(한글 포함) 지원**. 전 세계 표준 |

> 💡 기계는 2진수만 이해 → 문자를 코드 값으로 변환하여 저장  
> ASCII는 영어만 표현 → 다른 문자 사용을 위해 유니코드로 확장

- 실행 빈도수 표와 그래프 형태(가파르게 올라감) 이해 필수
- 입력 크기 n이 커질수록 복잡도 차이가 극명하게 나타남

## 📙 CH02 · 자료구조 구현 (포인터)

> 📌 **출제 범위:** 1차원 배열에서 포인터 연산 / 주소 연산 위주. 너무 꼬아서 내지 않음. **연습문제 풀이 참고.**

### 배열과 포인터 관계

- 배열은 내부적으로 **포인터 연산으로 변환**되어 저장됨
- `a[i]` = `*(a + i)` (같은 표현)
- **포인터 배열:** 포인터들의 배열 → `int *arr[5]`
- **배열 포인터:** 배열 전체를 가리키는 포인터 → `int (*ptr)[5]`

### 포인터 변수 기본 문법 (C언어)

```c
int a = 10;
int *p = &a;   // p에 a의 주소 저장

printf("%d", *p);   // 10 출력 (역참조)
printf("%p", p);    // a의 주소 출력

// 주소 연산
p++;    // 다음 int 크기(4바이트)만큼 주소 이동
p + n;  // n * sizeof(int) 만큼 이동
```

- `&` : **주소 연산자** (변수의 주소 반환)
- `*` : **역참조 연산자** (주소에 저장된 값 반환)
- 어떤 것의 **주소**인지, **무엇을 가리키는지** 구별 필수

## 📕 CH03 · 순차 자료구조 / 선형 리스트

### 순차 자료구조 vs 연결 자료구조 장단점

| 구분 | 순차 자료구조 (배열) | 연결 자료구조 (연결리스트) |
|:---:|:---|:---|
| **접근** | ✅ 인덱스로 O(1) 빠른 접근 | ❌ 순차 탐색 O(n) 느림 |
| **삽입/삭제** | ❌ 이동 필요 → 느림 | ✅ 포인터 변경만 → 빠름 |
| **메모리** | ❌ 고정 크기, 낭비 가능 | ✅ 동적 할당, 유연 |
| **오버헤드** | 없음 | ❌ 포인터 공간 추가 필요 |

---

### 1차원 / 2차원 배열 주소 계산 (★ 행 우선 출제)

#### 1차원 배열 주소

```
시작주소 + (인덱스 × 자료형 크기)

예: 시작주소 100, int(4byte), a[3]의 주소?
→ 100 + (3 × 4) = 112
```

#### 2차원 배열 — 행 우선(Row-major) 저장 (3차원 미출제)

배열 `A[m][n]`에서 `A[i][j]`의 주소:

```
시작주소 + (i × n + j) × 자료형크기

예: int A[4][5], 시작주소 200, A[2][3]의 주소?
→ 200 + (2 × 5 + 3) × 4
→ 200 + 13 × 4
→ 200 + 52 = 252
```

> 💡 PPT 11페이지 참고. 특정 값이 몇 번째에 저장되는지 **역계산 문제**도 연습할 것

---

### 선형 리스트 원소 삽입/삭제 이동 횟수 (PPT 20페이지)

| 연산 | 이동 횟수 공식 | 예시 (총 n=100, k번째) |
|:---:|:---:|:---|
| **삽입** | `n - k + 1` | 9번째 삽입 → 100 - 9 + 1 = **92번** |
| **삭제** | `n - k` | 9번째 삭제 → 100 - 9 = **91번** |

- 삭제 시 **오버헤드(쓸모없는 공간)** 발생 → 메모리 낭비
- 삽입: 뒤에서부터 밀어야 하므로 k번째 이후 원소 전부 한 칸씩 이동

---

### 희소행렬(Sparse Matrix) — 개선 방법 이해

| 방식 | 설명 |
|:---:|:---|
| **기존 방식** ❌ | 전체 행렬을 그대로 2차원 배열에 저장 → 0인 값이 대부분 = **메모리 낭비 심각** |
| **개선 방식** ✅ | 0이 아닌 값만 (행, 열, 값) 형태로 저장 → 메모리 절약 (PPT 27페이지) |

```
[전체 행 수, 전체 열 수, 0이 아닌 원소 개수]
[행 인덱스, 열 인덱스, 값]
[행 인덱스, 열 인덱스, 값]
...
// 소스코드는 몰라도 됨 — 개념과 저장 방식만 이해
```

---

### 다항식 선형 리스트 배열 저장 (PPT 34페이지)

- 다항식도 **배열로 저장 가능**
- 개선 방식: **차수(degree) + 계수(coefficient)** 쌍으로 저장
  - 예) `3x⁴ + 2x² + 1` → `[(4,3), (2,2), (0,1)]`
- 소스코드 불필요 — **간단 저장 가능하다는 개념만 이해**

## 📘 CH04 · 연결 리스트

> 🚨 **출제 포인트:** 단순/원형/이중 구조, 각 삽입·삭제·순회 알고리즘, 배열 vs 연결리스트 장단점 표. **노드 삽입·삭제 알고리즘 반드시 보기.**

### 단순 연결 리스트 (Singly Linked List)

```
HEAD → [data|link] → [data|link] → [data|link] → NULL
```

- 각 노드: **data + link(다음 노드 주소)**
- 마지막 노드의 link = **NULL**
- 한 방향으로만 탐색 가능
- **삽입:** 이전 노드의 link를 새 노드로, 새 노드의 link를 다음 노드로
- **삭제:** 이전 노드의 link를 삭제 노드의 다음 노드로 변경
- **순회:** HEAD부터 NULL 만날 때까지 link 따라 이동

---

### 원형 연결 리스트 (Circular Linked List)

```
HEAD → [data|link] → [data|link] → [data|link] ↩ HEAD
```

- 단순 연결 리스트와 동일하나, **마지막 노드의 link → 첫 번째 노드(HEAD)**
- NULL이 없음 → 끝없이 순환 가능
- **삽입/삭제/순회:** 단순과 유사하나, 마지막 노드 처리 시 HEAD로 연결 주의

---

### 이중 연결 리스트 (Doubly Linked List)

```
NULL ← [llink|data|rlink] ↔ [llink|data|rlink] ↔ [llink|data|rlink] → NULL
```

- 각 노드: **llink(이전) + data + rlink(다음)**
- **맨 앞 llink = NULL, 맨 끝 rlink = NULL**
- **양방향 탐색 가능** (쌍방 연결)

```c
// 이중 연결 리스트 삽입 (노드 X를 노드 A 뒤에 삽입)
X.llink = A;
X.rlink = A.rlink;
A.rlink.llink = X;   // 기존 다음 노드의 llink 변경
A.rlink = X;

// 삭제 (노드 X 삭제)
X.llink.rlink = X.rlink;
X.rlink.llink = X.llink;
```

---

### 배열 vs 연결 리스트 — 최종 장단점 비교표

| 항목 | 배열 (순차) | 연결 리스트 |
|:---:|:---|:---|
| **저장 방식** | 연속된 메모리 공간 | 노드 + 포인터 분산 저장 |
| **접근 속도** | ✅ O(1) 인덱스 직접 접근 | ❌ O(n) 순차 탐색 |
| **삽입/삭제** | ❌ O(n) 이동 필요 | ✅ O(1) 포인터만 변경 |
| **크기** | ❌ 고정 크기 (정적) | ✅ 동적 크기 변경 가능 |
| **메모리** | ❌ 낭비 가능성 | ❌ 포인터 공간 추가 필요 |
| **구현** | ✅ 단순 | ❌ 복잡 |

## ✅ 시험 전 최종 체크리스트

- [ ] 빅오 순위 암기 + for문 개수로 O(nⁿ) 판단
- [ ] 자료구조 분류표 (순차/연결 장단점) 빈칸 가능
- [ ] 존/팩 형식 개념 구분
- [ ] 2의 보수 쓰는 이유 (0 표현 유일, 범위 1개 더 많음)
- [ ] ASCII / 유니코드 특징 구분
- [ ] 2차원 배열 행우선 주소 계산 공식 적용
- [ ] 희소행렬 개선 저장 방식 (행, 열, 값)
- [ ] 단순/원형/이중 노드 구조 + 삽입/삭제 알고리즘

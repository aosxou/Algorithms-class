# 자료구조 시험 정리 📚

## 시간복잡도와 알고리즘 설계

### 빅오 표기법 (Big-O Notation)
- **최악의 경우(Worst Case)** 시간복잡도 표현
- **차수가 큰 항만 남긴다**
  ```
  O(3n² + 2n + 1) → O(n²)  (최고차항만)
  O(5n + 100)     → O(n)
  ```

### 시간복잡도 순위 ⏱️
```
빠름 ←─────────────────────────────────────────→ 느림
O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(n³) < O(2ⁿ) < O(n!)
```
- **지수(2ⁿ)**로 올라가면 어마무시하게 커짐! ⚠️

### for문 시간복잡도 계산
```c
for (int i = 0; i < n; i++)              // O(n)
    statement;

for (int i = 0; i < n; i++)              // O(n²)
    for (int j = 0; j < n; j++)
        statement;

for (int i = 0; i < n; i++)              // O(n³)
    for (int j = 0; j < n; j++)
        for (int k = 0; k < n; k++)
            statement;
```

### 알고리즘 성능 분석
- **실행 빈도수 표** 보기
- 각 문장 실행 횟수 세기
- 가파르게 올라가는 부분 체크

|------|------|------|------|

| 구분 | 순차 저장 (배열) | 연결 저장 (리스트) |
|------|-----------------|-------------------|
| **장점** | 접근 빠름 O(1) | 삽입/삭제 용이 |
| **단점** | 삽입/삭제 느림 | 접근 느림 O(n) |

### 수치 자료 표현

#### 1. 존(Zone) 형식 vs 팩(Pack) 형식
```
10진수: 237

존 형식: 0010 0011 0111
         (각 자릿수를 4비트로)
         → 사람을 위한 표현, 연산 불가
         → "back won" 같은 느낌 (의미는 있지만 연산 안됨)

팩 형식: 0010 0011 0111
         → 존 형식의 메모리 낭비 개선
```

#### 2. 2의 보수 (Two's Complement) ⭐
```
양수: 0111 1111 = +127
음수: 1000 0000 = -128

왜 2의 보수를 쓰는가?
1) 덧셈으로 뺄셈 가능: A + B의 보수 = A - B
2) 표현 가능한 값이 1개 더 많음
   - 1의 보수: -127 ~ +127 (256개 중 255개만 사용, 0이 2개)
   - 2의 보수: -128 ~ +127 (256개 모두 사용)
```

**8비트 표현 범위**
```
2^n을 기준으로 음수/0/양수 반으로 나누기
-2^(n-1) ~ 0 ~ 2^(n-1) - 1
-128 ~ 0 ~ 127
```

#### 3. 부동소수점 (Floating Point)
- **사용 이유**: 미세하고 세밀한 값까지 저장하기 위해
- 정수로는 표현 못하는 0.00001 같은 값 저장

### 문자 코드

#### ASCII (American Standard Code for Information Interchange)
- **7비트** 사용 (0~127)
- **최상위 비트**: 패리티 비트 (Parity Bit) - 에러 체크용
- **통신용** 표준 코드

#### Unicode
- **목적**: ASCII로 표현 못하는 다른 나라 문자 표현
- 한글, 한자, 이모지 등 포함

| 코드 | 비트 수 | 특징 |
|------|--------|------|
| **ASCII** | 7비트 + 패리티 1비트 | 영문, 숫자, 특수문자 |
| **Unicode** | 16비트 이상 | 전 세계 문자 |

## 2장: 배열과 포인터

### 배열과 포인터 관계
- **배열은 포인터 연산으로 변환되어 저장됨**
```c
int arr[5];
arr[i]  ==  *(arr + i)  // 포인터 연산
```

### 포인터 배열 vs 배열 포인터 ⚠️
```c
// 포인터 배열: 포인터들의 배열
int *ptr_arr[5];  // int* 타입 원소 5개를 담는 배열

// 배열 포인터: 배열을 가리키는 포인터
int (*arr_ptr)[5];  // 크기 5인 int 배열을 가리키는 포인터
```

### 포인터 연산 문제
- **어떤 것의 주소**를 묻는 문제
- **무엇을 가리키는지** 묻는 문제
- 포인터 변수 조작 문법 (연습문제 참고)

## 3장: 선형 리스트 (배열 표현)

### 주소 계산 공식 ⭐⭐⭐

#### 1차원 배열
```
Address(A[i]) = Base + (i × element_size)
```

#### 2차원 배열 (Row-major order)
```
Address(A[i][j]) = Base + ((i × 열크기 + j) × element_size)
```

**예제: 251이 어디에 저장?**
```c
int A[3][4];  // 3행 4열
// 행 단위 저장 시: A[0][0], A[0][1], ..., A[0][3], A[1][0], ...

값 251이 A[2][2]에 있다면?
→ (2 × 4 + 2) = 10번째 원소
```

#### 3차원 배열
```
Address(A[i][j][k]) = Base + ((i×면크기 + j×열크기 + k) × element_size)
면크기 = 행크기 × 열크기
```

---

### 선형 리스트 삽입/삭제

#### 삽입 시 이동 횟수
```
원소 100개, 9번째 위치에 삽입
→ 9~99번 원소를 10~100번으로 이동
→ 총 91개 이동

일반화:
n개 원소, k번째 삽입 → (n - k + 1)개 이동
```

#### 삭제 시 오버헤드
- 삭제 후 빈 공간 메우기 위해 원소 이동 필요
- **메모리 낭비** 또는 **이동 비용** 발생

---

### 희소행렬 (Sparse Matrix) 💡

**문제점**: 0이 대부분인 행렬 → 메모리 낭비 심함

**해결책**: 0이 아닌 값만 저장
```c
// 일반 저장
int matrix[1000][1000];  // 백만 칸 (대부분 0)

// 희소행렬 저장
struct SparseMatrix {
    int row;    // 행 번호
    int col;    // 열 번호
    int value;  // 실제 값
};
```

| 전체 행 | 전체 열 | 0이 아닌 개수 |
|--------|--------|--------------|
| 1000 | 1000 | 10 |

**27페이지**: 개선 방법만 이해하면 됨 (소스 코드 X)

---

### 다항식 표현

**일반 배열 저장**
```
3x⁴ + 2x² + 5
→ [3, 0, 2, 0, 5]  // 인덱스 = 차수
```

**개선된 저장** (34페이지)
```c
struct Term {
    int coef;   // 계수
    int exp;    // 차수
};

[{3, 4}, {2, 2}, {5, 0}]  // 0이 아닌 항만 저장
```

## 4장: 연결 리스트 (Linked List)

### 연결 리스트 종류

#### 1. 단순 연결 리스트 (Singly Linked List)
```
[data|next] → [data|next] → [data|next] → NULL
```

#### 2. 원형 연결 리스트 (Circular Linked List)
```
    ┌─────────────────────┐
    ↓                     |
[data|next] → [data|next] → [data|next]
```
- 마지막 노드가 첫 노드를 가리킴

#### 3. 이중 연결 리스트 (Doubly Linked List) ⭐
```
NULL ← [prev|data|next] ⇄ [prev|data|next] ⇄ [prev|data|next] → NULL
```

**구조**
```c
struct Node {
    int data;
    struct Node *llink;  // 왼쪽(이전) 노드
    struct Node *rlink;  // 오른쪽(다음) 노드
};
```

**맨 끝 원소**
- 데이터 있음
- `rlink = NULL`

---

### 배열 vs 연결 리스트 비교 ⭐⭐

| 항목 | 배열 (Array) | 연결 리스트 (Linked List) |
|------|--------------|--------------------------|
| **접근 시간** | O(1) - 빠름 | O(n) - 느림 |
| **삽입/삭제** | O(n) - 이동 필요 | O(1) - 포인터만 조작 |
| **메모리** | 연속적, 고정 크기 | 불연속적, 동적 할당 |
| **메모리 낭비** | 미리 할당 → 낭비 가능 | 필요할 때만 할당 |
| **구조** | 간단 | 복잡 (포인터 관리) |
| **캐시 효율** | 좋음 | 나쁨 |

---

### 노드 삽입/삭제 알고리즘 ⭐⭐⭐

#### 단순 연결 리스트 - 삽입
```
[A] → [C]에 [B] 삽입

1. new_node 생성
2. new_node->data = B
3. new_node->next = A->next  // B가 C를 가리킴
4. A->next = new_node         // A가 B를 가리킴

결과: [A] → [B] → [C]
```

#### 단순 연결 리스트 - 삭제
```
[A] → [B] → [C]에서 [B] 삭제

1. A->next = B->next  // A가 C를 가리킴
2. free(B)            // B 메모리 해제

결과: [A] → [C]
```

#### 이중 연결 리스트 - 삽입 (슈도코드)
```
[A] ⇄ [C]에 [B] 삽입

1. new_node->data = B
2. new_node->llink = A        // B의 왼쪽 = A
3. new_node->rlink = C        // B의 오른쪽 = C
4. A->rlink = new_node        // A의 오른쪽 = B
5. C->llink = new_node        // C의 왼쪽 = B

결과: [A] ⇄ [B] ⇄ [C]
```

#### 이중 연결 리스트 - 삭제
```
[A] ⇄ [B] ⇄ [C]에서 [B] 삭제

1. A->rlink = B->rlink   // A의 오른쪽 = C
2. C->llink = B->llink   // C의 왼쪽 = A
3. free(B)               // B 메모리 해제

결과: [A] ⇄ [C]
```

---

### 순회(Traversal) / 탐색(Search) 알고리즘

#### 단순 연결 리스트 순회
```c
Node *current = head;
while (current != NULL) {
    printf("%d ", current->data);
    current = current->next;
}
```

#### 원형 연결 리스트 순회
```c
Node *current = head;
do {
    printf("%d ", current->data);
    current = current->next;
} while (current != head);
```
